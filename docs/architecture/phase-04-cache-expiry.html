Cache 過期後會發生什麼?

使用者離開 2 小時後回來 — 4 個 breakpoint 全部失效的成本與恢復路徑
時間軸:從正常 → 閒置 → 重建
N
turn N — 正常對話中
使用者送出訊息,模型回應。cache 寫到當前 user message。
static · hit 0.1× session · hit 0.1× msg bp #2 · hit 0.1× msg bp #1 · write 1.25×
⌛
使用者離開 2 小時 …
messages 5m TTL → 5 分鐘後過期。
static / session 1h TTL → 1 小時後過期。
2 小時後,4 個 breakpoint 全部失效。
N+1
turn N+1 — 冷啟動
使用者回來繼續對話。所有 cache miss,要重寫一遍 prefix。 這一 turn 比較貴。
static · miss → write 2× session · miss → write 2× messages · miss → full 1× new bp #1 · write 1.25×
N+2
turn N+2 — 已恢復
cache 已重建。只要使用者在 TTL 內繼續對話,就回到正常 0.1× 讀取流程。
static · hit 0.1× session · hit 0.1× msg bp #2 · hit 0.1× msg bp #1 · write 1.25×
turn N+1 各 breakpoint 的成本
Breakpoint 內容 正常 (cached) 冷啟動 (miss)
system list[0] static 7 段 prompt 0.1× 讀 2× 寫↑1h TTL 寫入溢價
system list[1] session-stable 動態段 0.1× 讀 2× 寫↑1h TTL 寫入溢價
messages bp #2 整段對話歷史(讀取點) 0.1× 讀 1× 全價↑整段歷史按基礎價計
messages bp #1 當前 user message(寫入點) 1.25× 寫 1.25× 寫·本來就要寫,沒變化
最痛的是 messages — 長對話累積幾萬 token,正常時 0.1× 讀,冷啟動要付 1× 基礎價。 例如 50k token 的歷史,冷啟動多付 ~45k token 的差價(1× - 0.1×)。
為什麼不主動保溫(keep-alive)?
✗ 背景定時 ping
  • 每次續約至少付 0.1× 整份 prefix 的讀取費
  • 使用者可能根本不回來 → 白花
  • 需要排程基礎設施,複雜度增加
  • 大多數使用者會在合理時間內回來,N 次保溫 > 1 次重建
✓ 接受偶發重建
  • 只有真的回來那一 turn 才付成本
  • 使用者放棄會話 = 完全不花錢
  • 不需任何額外邏輯,Anthropic 端透明處理
  • 重建後立刻恢復長期 cache hit 流程
何時該調整 TTL?
使用模式
建議 TTL
理由
連續對話(每幾秒一句)
messages = 5m(預設)
下 turn 一定在 5m 內,寫入成本最低(1.25×)
間歇對話(10~50 分鐘一句)
messages = 1h
多付 0.75× 寫入,換取下 turn 仍 hit。否則每次都要重付全價歷史
久違回來(數小時/隔天)
維持預設,不調
1h 也救不了,反而每 turn 都付寫入溢價。接受偶發重建就好
同一 session 跨午休
static/session = 1h(預設)
系統段 1h TTL 足夠覆蓋一般工作間隔,messages 可能會過期但成本相對小
關鍵觀念 — Anthropic 只支援 5m 與 1h 兩種 TTL,沒有中間值。 調整的決策邊界很清楚:預期下次互動會在 5m 內?用 5m。會在 5m~1h 之間?用 1h。超過 1h?兩者都救不了。
