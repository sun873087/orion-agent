Compaction — 對話越長越要壓

cache 解決「重送」· compaction 解決「上限」· 兩者必須協作但會互相打架
兩道防線
Defense 1 autoCompact
主動 — 每 turn 進 API 前估 token,接近上限就提早壓。
1. 估 token(1 token ≈ 4 chars)
2. 若 used > max_context × 0.8
3. 取前 50% messages 摘要
4. 替換成 TombstoneBlock
5. 繼續送 API
Defense 2 reactive
被動 — autoCompact 估錯/沒救到時的最後一道保險。
1. 送 API → 抓 prompt_too_long 錯誤
2. 強制 compact(更激進的 ratio)
3. retry 一次
4. 仍失敗 → 回拋給使用者
TombstoneBlock — 摘要替換,不是刪除
壓縮前 · 10 messages · ≈ 50k tokens
u1 a1 u2 a2 u3 a3 u4 a4 u5 a5
↓ cutoff = 50%(index 0..4)
壓縮後 · 7 messages · ≈ 27k tokens
📜 Tombstone(原 5 條摘要 · ≈2k tokens) a3 u4 a4 u5 a5
Tombstone 內容 — SonnetSummaryStrategy 呼 LLM 寫成自然語言摘要, 保留:user 目標 / 關鍵工具呼叫結論(file paths, commands, findings)/ 決策 / 錯誤。 LLM 摘要失敗 → fallback 到 TruncateStrategy 純截斷加註記。

為什麼必要 — Anthropic API 強制要求 tool_use 與對應 tool_result 成對出現, 若 cutoff 切在中間,留下「孤兒 tool_use」會被 API 直接拒絕。函式偵測此邊界後自動把 cutoff +1,讓兩者一起進 tombstone。

與 cache 的衝突
動作 messages cache 結果 當下成本
對話累積(不 compact) rolling cache hit ≈ 100% 0.1× 讀 + 1.25× 寫(只新內容)
觸發 compact 前段結構變了 → prefix 失效 摘要 LLM 費 + 下 turn 重建 cache
compact 後 1~2 turn 逐步重建中 過渡期偏貴
compact 後 3+ turn cache 已恢復 回到 rolling cache 流程
為什麼門檻是 80% — 不是越早 compact 越好。compact 是昂貴事件(摘要費 + cache 重建), 要盡量延後到「非做不可」的時候,讓 cache 在 compact 之間盡可能長期生效。 80% 是「還來得及處理一次 compact + 後續幾個 turn」的安全餘量。
