Tool / MCP 大結果處理

Orion Agent · 大型 tool output 在進到 context window 之前如何被攔截、截斷、持久化

Tool 回傳值越大,context window 就越早撐爆、推理品質越差、token 成本越高。 Orion Agent 設計了三層防線:tool 出口硬截 → 統一持久化 + preview → context 達 80% 時自動 compaction。

§1 三層防線總覽

資料從 tool 到模型之間,會依序經過下面三道閘:任何一道夠用,後面就不需要動。

LAYER 1 · 早截
Tool 出口硬截
tools/*/
  • Bash / Grep / WebFetch 在執行時就限 size
  • Read / Write / Edit 超量 直接拒絕
  • 截掉的原文 不留快取
LAYER 2 · 持久化
≥100KB 寫檔 + preview
storage/tool_result.py
  • 寫到 sessions/<sid>/tool-results/
  • 模型只看到 2KB preview + envelope
  • MCP 與內建 tool 共用此路徑
LAYER 3 · 壓縮
Context 80% 自動 compact
compact/auto.py
  • 4 chars/token 保守估算
  • Sonnet 摘要前 50% messages
  • 摘要時 tool_result 再壓到 500 字

§2 內建 Tool 的執行端截斷 Layer 1

每個 tool 在執行階段自己負責限大小,把控制權壓在最早的環節。

Tool 上限 處理方式
Bashtools/shell/bash.py:30 30 KB 截尾,附 ... [output truncated at 30KB bytes]
Greptools/search/grep.py:24 50 KB / 500 筆 截尾,附 ... (truncated at 50KB bytes)
WebFetchtools/web/fetch.py:20 原始 200 KB / 輸出 50 KB 截尾,附 ... [+N chars truncated]
Readtools/file/read.py 256 KB 拒讀,回錯誤訊息,不進 context
Write / Edittools/file/{write,edit}.py 1 MB 拒寫,要求使用者拆檔

§3 統一持久化 Layer 2 · 內建 tool 與 MCP 共用

任何 tool 跑完後,core/tool_execution.py:309 都會 call maybe_persist_large_tool_result(),門檻 100 KB。

狀況 給模型的內容
空字串 (tool produced no output)
< 100 KB 原樣回填
≥ 100 KB 寫到 ~/.orion/sessions/<sid>/tool-results/<tool_use_id>.txt,模型只看到 envelope 
<persisted-output tool_use_id='toolu_abc123' path='~/.orion/sessions/<sid>/tool-results/toolu_abc123.txt' size=524288>
<前 2048 字 preview>
... [N more chars truncated, see persisted file]
</persisted-output>

模型若需要完整內容,可用 Read 或 Bash + jq 去開那個檔—— 它是「淡出 context 但仍可取回」,而非「永遠消失」。

§4 MCP 的完整流向

MCP 比一般 tool 多了 wrapper、binary、專屬 envelope 等設計, 但目前 runtime 實際走的還是和內建 tool 共用的 100KB 路徑。

1
MCP wrapper 內預壓 已串接
mcp/tool_wrapper.py:100-124
收集 type=text chunk 並 join。 image / resource 內容直接 elide 成 [image content elided — Phase 5 不展開,Phase 10 處理]。 完全沒 content 時 fallback 到 raw JSON 截 5000 字。
2
進入統一持久化 已串接
storage/tool_result.py · 100 KB 門檻
和內建 tool 走同一條 maybe_persist_large_tool_result()。 MCP 沒有獨立的 hot path。
3
MCP 專屬 envelope 設計但未啟用
mcp/large_output.py · 25K tokens (~100 KB) 門檻
設計帶 jq 範例提示的 envelope: jq '.content[0].text' <path>。 目前 process_mcp_result() 僅在 tests 出現,runtime 未呼叫。
4
MCP binary 持久化 helper helper 寫好但未串接
storage/mcp_output.py
persist_mcp_binary() 可寫 PNG/JPG/PDF 到 sessions/<sid>/mcp-binaries/ 並附 .meta.json sidecar (記 media_type / size / captured_at)。 但 wrapper 目前還是 elide 掉 image/resource,沒有 call 它。

換句話說:MCP text 大 → 走通用 100KB 持久化; MCP image/PDF → 目前完全 elide(只剩 placeholder 字串)。 Phase 10 之後 §3、§4 兩條路才會真正接起來。

§5 Context 層的最後一道:Compaction Layer 3

即便每個 tool 都規矩截斷,長對話中 tool result 仍會累積。Compaction 是最後保險。

觸發 / 機制 細節
auto_compact 觸發compact/auto.py:47-69 估算用量 = len(serialized) / 4 tokens。 超過 max_context × 0.8 就跑摘要 (可用 env ORION_AUTO_COMPACT_THRESHOLD 覆寫)。
摘要動作 把對話前 50% 餵給 Sonnet 壓成自然語言,失敗 fallback 純截斷。
tool_result 二次壓縮compact/strategies.py:62-69 摘要過程中遇到 tool_result content,只取前 500 字 + "...",避免摘要本身又被大 output 撐爆。
切點安全邊界 _adjust_cutoff_to_safe_boundary:絕不切在 assistant tool_use 跟其對應 tool_result 之間。

§6 設計取捨 & 注意事項

早截勝於晚收
寧可在 tool 出口就丟掉,也不讓大塊資料先進 context、再去摘要。Bash 30KB 與 Grep 50KB 都是這種哲學。
100KB 之上仍可取回
超過門檻的 tool result 寫到 session 目錄,模型可後續用 Read / Bash 重新打開——並非永久丟失,只是淡出 context。
沒有「自動 offload 機制」
對 Bash / Grep / WebFetch 的截斷部分 完全丟棄、無原始快取。要看完整輸出只能調小範圍重跑 (加 path / 用 head / 用 wc)。
Token 估算偏保守
用 4 chars/token,對中文偏高估、對英文偏低估。換言之 compaction 可能提早觸發,但不容易來不及。
MCP image/PDF 暫無內容
Phase 5 wrapper 直接 elide 成 placeholder 字串,Phase 10 才會接到 storage/mcp_output.py 的 binary 持久化。
MCP 專屬 jq envelope 未啟用
mcp/large_output.py 的 25K tokens 門檻 + jq hint 設計完整,但 runtime 還沒呼叫,目前 MCP text 共用通用 100KB envelope。
